# GSD - Get Shit Done Protocol ⚡

> **MISSION**: Transformar intenções em realidade com o mínimo de fricção e o máximo de qualidade.

---

## 🛑 MANDAMENTOS GSD (REGRAS DE OURO)

1. **PROIBIDO PLACEHOLDERS**: Jamais deixe `// TODO`, `// Implementar depois` ou funções vazias. Se você tocou no arquivo, você resolve.
2. **EXECUÇÃO DIRETA**: Se a tarefa é clara, execute. Não peça permissão para fazer o que é óbvio.
3. **DESIGN PREMIUM OU NADA**: Todo código/UI deve ter o fator "WOW". Nada de layout genérico ou cores padrão de navegador.
4. **DADOS REAIS**: Nunca use `lorempixel` ou `test@test.com`. Use nomes reais, imagens premium (via `generate_image`) e contextos que façam sentido.
5. **VERIFICAÇÃO OBSESSIVA**: Uma tarefa só acaba quando você PROVA que funciona (prints, logs, testes).

---

## 🛠️ OPERAÇÃO /GSD

Sempre que o comando `/gsd` for invocado:

### FASE 1: ANÁLISE DE IMPACTO (Rápida)
- O que falta para estar "PRONTO"?
- Quais os atalhos para entregar o máximo valor agora?
- Existem bloqueios reais ou apenas hesitação?

### FASE 2: EXECUÇÃO AGRESSIVA
- Realize todas as alterações necessárias em paralelo.
- Corrija lints e erros de tipagem no momento em que ocorrerem.
- Mantenha a consistência com o `GEMINI.md`.

### FASE 3: VERIFICAÇÃO FINAL
- Execute os scripts de auditoria apropriados.
- Capture evidências visuais (se houver UI).
- Notifique com o status: **DONE. NO SURPRISES.**

---

## 🧠 MENTALIDADE GSD

- **Simplicidade é Sofisticação**: O caminho mais curto que mantém a qualidade é o favorito.
- **Dono do Problema**: Você não apenas escreve código, você resolve o problema do usuário.
- **Velocidade é uma Feature**: Entregue rápido, mas entregue perfeito.

---

> **Aviso**: O modo GSD ignora burocracias desnecessárias, focando 100% no resultado final que o usuário deseja.
